# GOTCHAS.md — 実装上の落とし穴・注意点

> 着手前に必読。特に「データソースの現実」を誤解すると Phase 1 で手戻りが大きい。

## 1. 【最重要】特許情報取得API は「横断検索」できない
特許庁の **特許情報取得API** は意匠も対象だが、**出願番号を指定して経過情報・書類実体を取得する REST API**であり、
「意匠分類 × 企業 × 公報発行日（直近1〜2年）」での**横断検索（discovery）はできない**。

- 用途は「既知の出願番号の**補完（enrichment）**」。
- 意匠の書誌情報は **2000年1月以降**、書類実体（図面等のHTM）は **2019年1月以降**のみ。
- **利用者登録（無料）＋利用申込（メール）が必須**、ID/APIごとに**日次アクセス上限**あり（トークンは発行後1時間有効）。

→ したがって：
- **Phase 0（MVP）はサンプルデータで完結**させる（このAPIに依存しない）。
- 本アプリの「分類・企業・期間で探す」横断検索が必要になる **Phase 1 では、特許情報標準データ／一括ダウンロードから自前の軽量データを生成**するのが正攻法。取得APIはあくまで番号ベースの補完に使う。
- Phase 1 の一括データ処理はブラウザ内で行わず、オフライン処理またはビルド処理で軽量データを生成する。
- 申込・規約・仕様は API 情報提供サイト（`ip-data.jpo.go.jp`）で都度確認すること（試行提供のため条件が変わりうる）。

## 2. J-PlatPat の Web UI 自動取得は採用しない
本プロジェクトではJ-PlatPatのWeb UIを用いた自動取得を採用しない。画面変更に弱く、アクセス負荷、利用条件、著作権・再利用条件の確認が必要になるためである。

- J-PlatPat に公開検索APIが無いことだけを理由に断定的な法的評価をしない。
- 横断検索の実データは §1 のとおり「一括DL→オフラインまたはビルド処理で軽量データ生成」で対応する。

## 3. WEB/プレス/新聞/株主総会は「準備中」固定
- 依頼主の明示指示。**取得処理を実装しない。** UI は disabled＋「準備中」バッジのみ。
- これらは利用許諾契約・著作権の整理が前提で、軽量MVPの趣旨に反する。

## 4. 意匠の「公開時期」と日付フィールドの定義
- 意匠と特許の公開時期を一律に比較する断定表現を避ける。
- 表現は **「案件によっては特許の出願公開より早期に把握できる可能性がある」** に統一する。
- Phase 0 の期間フィルタ（直近1年/2年）は、サンプルデータの `dataAsOf` を基準に計算する。
- 各レコードの日付フィールドは **`gazetteDate`（公報発行日）** に固定し、UI/ロジックで定義を一貫させる（出願日・登録日と混在させない）。
- 意匠には**秘密意匠**等の制度上の例外があるため、将来実データ接続時は「いつ公衆が知り得たか」の扱いに注意（Phase 1 で検討。Phase 0 はデモ用サンプルデータとして明示する）。

## 5. サンプルデータは実データと誤認させない
- 画面に **デモ用サンプルデータ** と **ルールベース分析** を常時表示する。
- Phase 0 の意匠情報ラベルは **デモ用意匠情報** とする。
- 各 `DesignRecord` には `sourceLabel` を持たせ、Phase 0 では **デモ用意匠情報** を設定する。
- `generatedBy` は Phase 0 では常に `'rules'`。

## 6. 図面・ロゴの権利関係
- **実在製品の意匠図・企業ロゴをリポジトリに同梱しない。**
- サンプルの `imageRef` はプレースホルダ／自作の抽象図に限定。

## 7. 静的サイトと LLM 呼び出し
- Phase 0 では LLM 連携、APIキー入力、`USE_LLM` による切替を実装しない。
- APIキーをフロント/リポジトリにハードコードしない。`.env` をコミットしない。
- 将来 LLM を使う場合も、静的サイトのまま安全に扱えるキー管理、コスト、安全なJSON検証、利用規約を別途整理してから実装する。
- Phase 0 で残す将来拡張点は `AnalysisEngine` インターフェースのみ。

## 8. GitHub Pages の base パス
- リポジトリ配下デプロイ（`https://<user>.github.io/<repo>/`）では、Vite の `base` 設定やアセットの相対パスを誤ると白画面になる。
- 既存 GENBA Search のデプロイ設定がリポジトリ内に存在する場合はそれに合わせる。
- 既存設定が無い場合は、リポジトリ名配下デプロイとユーザー/組織ページ直下デプロイのどちらかを実装前に確認する。

## 9. 文字コード・日本語
- すべて UTF-8。日本語ラベルは定数に集約し、文字化け・全角半角の混在に注意。

## 10. 出力はあくまで「参考情報」
- 知財戦略コメントは**法的助言ではない**。画面に注記を常設し、断定を避けた表現にする（特に知財・出願関連）。
- 各分析示唆には `evidenceIds`、`metric`、`confidence` を付与し、文章だけのブラックボックス出力にしない。
- `confidence` は法的確度ではなく、サンプルデータ内の根拠量・一貫性に基づく分析上の目安として扱う。

## 11. スコープ・クリープ防止
- 「ついでに検索結果一覧」「ついでにPDF出力」等を勝手に足さない。
- SPEC の①〜⑤と結果カードの構造を厳密に再現することが Phase 0 のゴール。
- Phase 0 では `RuleBasedAnalysisEngine` に集中し、LLMや実データ取得の実装へ広げない。
